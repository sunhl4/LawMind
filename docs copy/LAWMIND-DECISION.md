# LawMind 决策文档

本文档记录一个关键战略决策：**LawMind 不做普通 OpenClaw 插件，而做独立核心产品；同时有选择地借鉴和复用 OpenClaw 的成熟基础能力。**

这是后续架构、研发边界和商业化路径的统一依据。

---

## 一、结论

**最终路线：独立内核 + 适配层，而不是插件模式。**

具体来说：

- **不选插件寄生**：LawMind 的差异化不只是多几个法律工具，而是要重构信任、审批、审计、记忆和交付链路。
- **不做纯从零重写**：OpenClaw 在 agent loop、工作区记忆、消息通道、工具机制和运行时组织上已经有很多成熟经验，值得借鉴。
- **选择中间路线**：LawMind 拥有自己的核心协议、状态、审计和产品层；OpenClaw 只作为参考实现或可选适配层。

---

## 二、为什么不做 OpenClaw 插件

### 1. 插件适合“加能力”，不适合“改默认行为”

插件适合增加：

- 新工具
- 新模型接入
- 新消息渠道
- 新技能和模板

但 LawMind 需要改的是：

- 信任模型
- 执行与审批模型
- 审计与责任边界
- 记忆契约
- 文书交付协议

这些都属于系统级约束，不应该依赖宿主是否“配合”。

### 2. 法律行业需要自己掌控责任边界

律师场景不是一般办公场景。AI 的错误可能直接转化为：

- 客户损失
- 证据风险
- 违约风险
- 执业风险

如果核心行为仍由宿主系统定义，LawMind 很难对外承诺稳定的合规边界。

### 3. 产品护城河不在插件数量，而在产品默认值

真正构成 LawMind 护城河的，是这些默认能力：

- 高风险动作必须确认
- 文书输出前必须审阅
- 检索结果必须带来源
- 关键规则不靠上下文临时记忆
- 案件与律师偏好有独立记忆层

这些都应该是产品核心，而不是宿主环境里的一个可选扩展。

---

## 三、为什么也不做完全从零重写

OpenClaw 已经验证了很多底层问题：

- agent loop 怎么组织
- 工作区记忆如何落到 Markdown
- 工具如何分组与控制
- 会话、消息通道、模型与扩展如何拼起来

因此 LawMind 没必要重复发明这些基础件。

更合理的做法是：

- 借鉴其设计
- 复用其成熟思路
- 仅在需要强掌控的地方自建核心

这可以把团队精力集中在真正有壁垒的部分：法律工作流、审计链、双记忆、模型路由和文书交付。

---

## 四、LawMind 的系统边界

### 必须由 LawMind 自己掌控的部分

1. **任务协议**
2. **双记忆契约**
3. **模型路由与结果合并**
4. **两阶段执行与人工确认**
5. **审计事件流**
6. **Word/PPT 交付引擎**

### 可以借鉴或适配 OpenClaw 的部分

1. **工作区与 Markdown 记忆哲学**
2. **memory search / memory get 一类能力设计**
3. **消息渠道接入经验**
4. **CLI 与配置组织方式**
5. **沙箱、审批与工具策略的实现思路**

---

## 五、目标架构决策

LawMind 采用三层结构：

### 1. LawMind Core

负责：

- 双记忆管理
- 模型路由
- 检索编排
- 草稿生成
- 审批流
- 审计流
- 文档交付

### 2. Runtime Adapters

负责把外部世界接入 LawMind Core，例如：

- OpenClaw adapter
- 邮件 adapter
- 企业 IM adapter
- HTTP/API adapter

### 3. Artifact Layer

负责模板和交付：

- 法律检索报告
- 律师函
- 客户汇报 PPT
- 后续其他法律文书

这层只消费结构化草稿，不直接参与模型推理。

---

## 六、第一阶段产品范围

第一阶段只做最小闭环，不做大而全：

1. 双记忆文档
2. 检索与结构化整理
3. Word 文档生成
4. 人工审核点
5. 审计日志

PPT、案件级深度记忆、插件市场和高级协作都放到后续阶段。

---

## 七、执行原则

- **先定协议，再写功能**
- **先做 Word，再做 PPT**
- **先做审计，再做更多自动化**
- **先做独立核心，再接宿主运行时**
- **先把高风险行为约束住，再追求“更像人”的智能**

---

## 八、一句话决策

**LawMind 是独立产品，不是 OpenClaw 插件；但它会站在 OpenClaw 的肩膀上，把成熟的通用能力转化为律师行业可落地、可审计、可商业化的系统。**
